# 02. 운행 (❄️v4.10)

## (1) 운행 준비

> 온보딩 완료 후 배차 대기 상태로 진입하기 전 본인의 계정 및 차량 정보를 확인할 수 있다. 

### A. 화면 구성

![img](images/02_운행준비.png)

#### a. 메뉴

- 운행 준비 상태에서 메뉴에 진입할 수 있다. 
- 운행 중 메뉴 조작으로 인한 안전 문제를 방지하기 위해 운행 준비 상태에서만 메뉴 진입이 가능하도록 제한한다. 

#### b. 계정 확인 정보

- 본인 계정 정보를 확인할 수 있다. 
- 계정 확인 정보 :`프로필사진` `실명`

#### c. 차량 정보 

- 기사 정보에 매핑된 차량 정보를 확인할 수 있다. 
- `차량 번호` 만 노출하고 별도 차량 정보 변경 동선은 고려하지 않는다.

#### d. 콜 받기 시작

- 콜 받기 시작 선택 시 배차 대기 상태로 전환된다. 

#### e. 서비스 운영 시간 아님

- [운행 준비] 상태에서 [배차 대기]로 전환하는 시점에 서비스 운영 시간 여부를 체크한다. 
- 서비스 운영 시간은 지역 별로 하드코딩되어 있는 값을 임시로 활용한다 
- 운영 시간이 아닐 때에는 [배차 대기]로 전환은 되나, 부가의 서비스 운영 시간 안내 팝업을 제공한다. 

![img](images/02_운영시간체크.png)


## (2) 배차 대기

> 본격적으로 콜 매칭이 가능한 상태로 콜 매칭이 활발하게 진행 중임을 확인할 수 있다. 오픈 이후 해당 화면에서 콜 매칭이 활발하게 진행되고 있음을 보여줄 수 있는 지표를 추후 적용한다.

### A. 화면 구성 (🎄v4.9.5)

![img](images/02_배차대기.png)

#### a. 배차 대기 안내(🎄)

- 현재 배차 대기 상태임을 인지할 수 있도록 상태 정보를 제공한다. 
- 활성화 정보 디스크립션에 제공 
  - `{운행지역명}` 승객들이 이동을 준비하고 있습니다.(🎄)
  - {운행지역명}은 서버에서 내려주는 기사 정보의 District 값의 {도 이름}+{시 이름}을 참조한다.
  - [예외] Distict 값이 빈 경우에는 {운행지역명}을 미노출한 나머지 안내 문구를 제공한다.
  

#### b. 콜 멈추기

- 콜 멈추기 선택 시 운행 준비 상태로 전환되며, 배차가 불가능하다

#### c. 닫기 버튼

- 콜 만료 시 안내 화면은 2s 간 한시적으로 노출되나, 그 전에 화면을 닫고 싶을 경우를 대비해 닫기 버튼을 제공한다, 

#### d. 배차 완료 문구 노출

- 콜 안내 화면에서 배차 대기 상태로 자동 전환되는 경우, 한시적으로 배차 완료 문구를 노출한다. 
- 배차 대기 상태 자동 전환 케이스 :
  - timeout 내 배차 수락/거절하지 않았을 때 
  - 다른 기사가 해당 콜을 먼저 수락했을 때 
  - timeout 이전 demand가 취소되었을 때
  
- 노출 시간 : 2s
  - 노출 시간은 테스트 후 조정 예정 

#### e. 화면 켜짐 유지 

- 배차 대기(READY) 상태 && 서비스 화면이 포그라운드로 노출되고 있는 경우에는 화면 꺼짐을 제한한다.




### B. 배차 정책 (🎄v4.9.5)

#### a. Constraint (🎄)

- 서비스 타입 : 호출 승객이 선택한 서비스 타입과 같은 서비스 타입을 운행 중인 차량을 탐색한다. (🎄)
- 최대 검색 범위 : 콜의 출발지 기준 해당 서비스 타입의 최대 직선거리 내에서 차량을 탐색한다.
  - 최대 직선거리: 서비스 타입별 `{최대 배차거리}`값 참고(🎄)

- Timeout 3분 : 하나의 콜은 최대 3분까지 매칭을 시도하고, 실패 시 매칭 실패 처리한다.

#### b. 매칭 로직 (🎄)

- 서비스 타입이 일치하며 (🎄), 호출 시점에 최대 검색 범위 내에 있는 차량을 pool로 관리한다 
  - 복수개의 demand가 동일한 차량 pool을 공유할 수 있다.

- 차량 pool 중 Score가 가장 높은 순서대로 배차 Q를 생성한다. 
  - 오픈 시점 Score 기준은 `기사 위치` ~ `출발지`의 직선 거리이다.
  - 배차 Queue는 5초마다 업데이트한다.
  - 단, 다른 demand의 콜안내가 발생한 차량의 경우 배차 Q에서 제외된다. 
- 매 업데이트 시점마다 배차 Q 중 Score가 높은 차량 3대에게 콜 안내를 제공한다. 
- 콜 안내를 받은 차량 중 거절 이벤트가 발생할 경우 배차 Q에서 즉시 제외된다.
- 콜 안내는 최대 5초까지 유지되며, 5초 내 거절/수락하지 않는 경우 자동 만료 처리된다. 

#### c. 매칭 실패 케이스

- Timeout 3분 : 타임아웃 시점까지 매칭 실패 시 

- 3분 전 매칭 실패 : pool에 있는 모든 차량이 배차를 거절하거나 timeout되었을 때



## (3) 콜 안내

> 배차 대기 상태에서 매칭 성공 했을 때 매칭된 호출 건에 대한 안내를 제공한다. 해당 안내 정보를 확인하고 사용자는 수락/거절 여부를 판단할 수 있다. 

### A. 화면 구성 (❄️v4.10)

![img](images/02_콜안내.png)

#### a. 콜 거절 버튼

- 거절 시 바로 배차 대기 상태로 전환된다. 

#### b. 현 위치~출발지 거리 정보 (❄️v4.10)

- 사용자는 해당 콜이 현위치에서 얼마나 멀리 떨어져 있는지 확인할 수 있다. 
- 오버롤 라우팅 계산 값을 기준으로 노출한다. (❄️)
- 거리 단위는 km 단위로 소숫점 1자리까지 반올림해서 노출한다. 
- 단, 실 직선 거리가 0.1km 미만인 경우 일괄적으로 거리 정보는 '100m 이내'로 표기된다

#### c. 타이머

- 콜 안내 생성 시점부터 5초 카운트 
- 0초 도래 시 콜안내 화면 자동 삭제 처리 

#### d. 출발지 - 목적지 정보 

- 라이더앱/운영툴에서 발생한 호출의 `POI명` + `도로명 주소` + `좌표`를 전달받는다. 
- 그 중 `POI명` + `도로명 주소`를 노출한다.
- 도로명 주소는 `Si_name`을 포함한 하위 주소를 노출한다. (예 : 구리시 동구릉로136번길 57)
  - 정책 결정 근거 : 
    - 시 단위부터 노출해도 대략적인 위치를 판별할 수 있다. 
    - 관외 이동 시, 원거리 이동보다 인접한 도시 간 이동이 많을 것이다
    - 동명의 시 이름이더라도 예상 이용 요금 정보과 함께 제공된다면, 위치 판별이 가능할 것이다
  - [참고] 셔클의 주소 체계
    - 도/특별시/광역시 : do_name (1depth)
    - 시/군/구 : si_name (2depth)

##### POI / 도로명 주소 노출 상세 정책 

- 라이더앱과 동일한 노출 정책을 가져간다
  -  `POI명` : POI > 도로명 displayname > 도로명 fullname > 지번 displayname > 지번 fullname 중 1
    - 단, POI가 없거나, 도로명 또는 지번 displayname도 없을 경우 full name으로 모두 제공함
  - `도로명 주소`: 도로명 fullname > 지번 fullname 중 1

#### e. 예상 이용 요금 정보 (❄️v4.10)

- 예상 이용 요금
  - 택서타별 요금 체계에 따라 노출한다. (🎄🎄)
    - 의왕: `taxiFare` +  `봉사료({호출 요금})`노출
      - taxiFare: 출발지를 `출발지 좌표`, 도착지를 `목적지 좌표`로 적용했을 때 Tmap API 결과에서 `taxiFare` 값 참조
      - 봉사료({호출 요금}): 관제에 등록된 서비스 타입 종속 호출 요금
    - 영암: `거리 기준 요금` + `기본 요금({호출 요금})` 노출
      - 거리 기준 요금: 바우처에 설정된 요금 값 (마을간 거리에 따라 기 등록되어 있는 고정값)
      - 기본 요금({호출 요금}): 관제에 등록된 서비스 타입 종속 호출 요금
    - 영덕 : `taxiFare` 노출 (❄️)
      - taxiFare: 의왕과 동일
- 호출 요금 
  - `호출 요금`이 있는 경우, 택시 서비스 타입별 `호출 요금명`  및 `호출 요금`노출
  - `호출 요금`이 없는 경우, 미노출 (❄️)


#### f. 콜 수락 버튼

- 수락 시 `03.콜 > 출발지 안내 `화면으로 전환된다. 



### B. 백그라운드 앱 실행 중 콜 안내 동작

![img](images/02_백그라운드콜안내.png)

- 앱이 백그라운드에서 실행 중일 때 (포그라운드에 서비스 화면이 미노출되나 실행 중일 때) 배차에 성공할 경우, 다음과 같이 동작한다
  - 플로팅 버튼이 콜 발생 중 타입으로 변경
  - 콜 카드 노출된 상태로 앱 서비스 화면이 포그라운드로 전환
- '거절' 선택 시
  - 콜 거절 처리 
  - 앱 서비스 화면이 백그라운드 전환
- '수락' 선택 시
  - 콜 수락 처리 
  - '출발지로 길안내' 화면 전환
- 콜 카드 타임아웃 시 (= '거절'과 동일하게 처리)
  - 콜 거절 처리 
  - 앱 서비스 화면이 백그라운드 전환
- 승객 > 콜 취소 시
  - 서비스 포그라운드 유지한 채로 콜 대기 중(READY) 화면으로 전환
  - 콜 취소 팝업 노출



## (4) 차량 위치 공유 

- 원할한 서비스 제공을 위해 상태값에 따라 주기적으로 차량의 위치를 서버에 공유한다

| 상태        | 공유 목적                          | 공유 주기 |
| ----------- | ---------------------------------- | --------- |
| 운행 준비   | 운영툴 내 차량 위치 확인을 위해    | 1s        |
| 배차 대기   | 호출 매칭을 위해                   | 1s        |
| 콜 안내     | 호출 매칭을 위해                   | 1s        |
| 출발지 안내 | 라이더앱에서 차량 위치 파악을 위해 | 1s        |
| 목적지 안내 | 운행 경로 파악을 위해              | 1s        |



## (5) 예외 케이스 정의

### A. 차량 정보 없음 

#### a. 발생 경로 

- 운영툴에서 사용자가 점유한 차량을 삭제하는 경우 발생 가능 
- 법인 기사만 해당

#### b. 차량 정보 갱신 시점 

- 앱 로그인 시 체크 (자동 로그인 포함)
- 메뉴 > 내 정보 진입 시점 

#### c. 차량 정보 변경 플로우

- 차량 정보 갱신 시점에 차량 정보가 없다면, 차량 정보를 변경할 수 있는 동선을 제공한다

![img](images/01_차량없음정보변경.png)

#### d. 콜 받기 제한 플로우

- 차량 정보가 없는 경우, 다음 두 동선에서 팝업을 노출함으로서 사용자에게 차량 정보 등록이 필요한 상태임을 고지한다. 
  - 운행 준비 상태에서 배차 대기 상태로 전환하는 시점
  - (자동)로그인 직후 운행 준비 화면에 랜딩하는 시점 

- 배차 대기 상태 (SupplyState = ON)일 때는 운영툴 내 차량 삭제가 불가하다.

![img](images/01_차량없음콜제한.png)
